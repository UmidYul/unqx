## План: Pixel-Perfect Копирование Next Дизайна в `express-app` (поэтапно, постранично)

### Краткое резюме
Цель: сделать визуал `express-app` **точь‑в‑точь** как текущий Next UI для всех страниц и состояний.  
Эталон: **текущий Next код**.  
Приёмка: **screenshot-diff** на `390x844`, `768x1024`, `1440x900` с порогом `<= 0.2%`.

---

### Объём (зафиксирован)
1. Обязательные страницы:
1. `/`
2. `/:slug` (активная визитка)
3. `/:slug` (unavailable)
4. `404/not-found`
5. `500/error`
6. `/admin`
7. `/admin/dashboard`
8. `/admin/cards/new`
9. `/admin/cards/:id/edit`
10. `/admin/stats`
11. `/admin/logs`

2. Обязательные состояния:
1. Hover/focus/disabled для кнопок и полей.
2. QR modal открытое состояние.
3. Empty states в таблицах.
4. Ошибки форм (login/card form).
5. Анимации (`animate-in`, `shimmer`) и их тайминги.
6. Mobile/tablet/desktop адаптив.

---

### Важные изменения API/интерфейсов
1. Публичные URL и JSON-контракты API не меняются.
2. Добавляются только внутренние артефакты приёмки дизайна:
1. Скрипты визуального сравнения.
2. Эталонные скриншоты Next и сравнения Express.
3. Документация по визуальной приёмке.
3. Типы/схемы данных и бизнес-логика остаются без изменений.

---

### Поэтапная реализация (decision-complete)

### Этап 0 — База сравнения и стабильность рендера
1. Подготовить детерминированный визуальный контур:
1. Один фиксированный seed карточки (`AAA001`) с аватаром, тегами, кнопками, заполненными контактами.
2. Фиксированная timezone `Asia/Tashkent`.
3. Одинаковый браузер/UA (Chromium) для Next и Express.
4. Маскировать или стабилизировать динамические зоны (например, год в футере, текущее время в логах), чтобы не ловить ложные diff.
2. Поднять два окружения одновременно:
1. Next эталон.
2. Express кандидат.
3. Ввести единый naming для скриншотов:
1. `next/<route>/<state>/<viewport>.png`
2. `express/<route>/<state>/<viewport>.png`
3. `diff/<route>/<state>/<viewport>.png`

### Этап 1 — Глобальный слой дизайна
1. Сверить и привести к 1:1:
1. Типографику (Inter, веса, размеры, line-height, tracking).
2. Базовые цвета, прозрачности, blur, радиусы, тени.
3. Контейнеры, отступы, вертикальный rhythm.
4. Глобальные keyframes (`shimmer`, `fadeSlideUp`) и длительности.
2. Зафиксировать таблицу дизайн-токенов с числами из Next классов.

### Этап 2 — Страница `/` (Home)
1. Перенести структуру блоков 1:1 из `app/page.tsx`.
2. Воссоздать фоновые blur-формы, метрики, карточки преимуществ, секцию “Как это работает”, footer.
3. Сверить:
1. Размеры заголовков по брейкпоинтам.
2. Межблочные интервалы.
3. Цвета/прозрачности блоков.
4. Приёмка этого этапа: screenshot-diff `<=0.2%` на 3 viewport.

### Этап 3 — Публичная визитка `/:slug` (главный критичный экран)
1. Перенести DOM-структуру и позиционирование максимально близко к `app/[slug]/page.tsx`.
2. Полностью повторить:
1. SVG lightning (форма + все позиции desktop/mobile + углы/размеры/повороты).
2. Размеры карточки, отступы, типографику, verified icon.
3. Теги, кнопки, shimmer, footer блок About info.
4. Анимационные задержки для элементов (stagger).
3. Сверить интерактив:
1. Copy phone button визуал и состояния.
2. Share/VCF кнопки.
4. Приёмка: отдельные сравнения для заполненной карточки и карточки с минимальным набором полей.

### Этап 4 — Публичные состояния unavailable / not-found / 500
1. Подогнать карточки состояния по размеру, бордерам, типографике, вертикальному центру.
2. Проверить одинаковую визуальную плотность и контраст текста.
3. Приёмка: 3 viewport, все состояния.

### Этап 5 — Admin layout + `/admin` login
1. Хедер protected layout:
1. Высота, отступы, nav pills, активные/hover состояния.
2. Кнопка logout.
2. Страница login:
1. Контейнер, бордер, шрифт, поля, кнопка, ошибка.
3. Приёмка: normal + error state.

### Этап 6 — `/admin/dashboard` + QR modal
1. Таблица и фильтры:
1. Колонки, paddings, header стиль, empty row.
2. Статус pill active/inactive.
3. Action buttons.
2. QR modal:
1. Размер модалки, overlay, отступы, кнопки.
2. QR блок и встроенный logo (`/brand/unq-mark.svg`) как в Next.
3. PNG/SVG download кнопки визуально 1:1.
3. Приёмка: базовый экран + modal open state.

### Этап 7 — `/admin/cards/new` и `/admin/cards/:id/edit` (самый объёмный этап)
1. Повторить секции формы из `CardEditor`:
1. Основные данные.
2. Аватар upload/crop.
3. Теги-ссылки (sortable rows).
4. Кнопки (sortable rows + toggle).
5. Контакты.
6. Статистика блока edit.
7. Action panel.
2. Визуально выровнять:
1. Бордеры секций, row-сетки, кнопки удаления, drag-handle.
2. Cropper контейнер/preview.
3. Ошибки валидации и request errors.
3. Приёмка:
1. New page.
2. Edit page (with stats).
3. Form with validation errors.
4. Avatar crop visible state.

### Этап 8 — `/admin/stats`
1. Карточки KPI 1:1.
2. График визуально под Next (линии, цвета, сетка, легенда, paddings).
3. Топ-10 таблица.
4. Приёмка: populated + fallback empty series.

### Этап 9 — `/admin/logs`
1. Фильтр, таблица, пагинация, cleanup action состояние.
2. Состояния:
1. Есть логи.
2. Пусто.
3. Cleanup success message.
3. Приёмка: 3 viewport.

### Этап 10 — Полировка и финальный прогон
1. Полный regression screenshot sweep по всем страницам/состояниям.
2. Устранить остаточные расхождения `>0.2%`.
3. Проверка кликабельных интерактивов после финального CSS-тюнинга.
4. Обновить документацию в `express-app/README.md`:
1. Как запускать visual compare.
2. Как читать diff и принимать экран.

---

### Тест-кейсы и сценарии приёмки
1. Visual tests:
1. Каждый экран и состояние в 3 viewport.
2. Порог расхождения `<=0.2%`.
2. Functional smoke после каждого этапа:
1. Основная навигация страниц.
2. API-driven рендер карточек/таблиц.
3. Форма create/edit и avatar upload.
4. QR modal open/close/download.
3. Accessibility sanity:
1. Фокус видим на интерактивах.
2. Модалка закрывается корректно.
3. Контраст текста не ухудшен относительно эталона.

---

### Явные предположения и дефолты
1. Источник правды дизайна: текущий Next UI в репозитории.
2. Объём: все страницы + все ключевые состояния.
3. Приёмка: screenshot-diff (не только ручной просмотр).
4. Viewport matrix: `390x844`, `768x1024`, `1440x900`.
5. Допустимый итоговый diff: `<=0.2%`.
6. Если конфликт между “красиво” и “как в Next”, выбирается “как в Next”.
7. Любые изменения API/бизнес-логики вне визуала считаются вне scope.
